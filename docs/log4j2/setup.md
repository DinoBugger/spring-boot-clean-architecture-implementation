# Log4j2 Setup Guide

This document explains how Log4j2 is configured in this Spring Boot project and how you can use it after cloning the repository.

---

## 1. What changed

Spring Boot uses Logback by default. In this project, the logging backend was switched to **Log4j2**.

The main reasons for using Log4j2 are:

- cleaner and more flexible logging configuration
- easy control over package log levels
- consistent console output during development and tests

---

## 2. Gradle changes

The logging setup is controlled in `build.gradle`.

### Added

- `org.springframework.boot:spring-boot-starter-log4j2`

### Removed from the default logging path

The build excludes Spring Boot's default logging starter:

- `org.springframework.boot:spring-boot-starter-logging`

This is important because Spring Boot should only use **one** logging backend at runtime.

### Result

After the change, the application will start with Log4j2 instead of Logback.

---

## 3. Logging configuration file

The runtime logging configuration is stored in:

```text
src/main/resources/log4j2-spring.xml
```

This file is loaded automatically by Spring Boot when Log4j2 is present on the classpath.

### What it contains

- a console appender
- a shared log pattern
- log level settings for:
  - your application package: `com.dinobugger.cleanproject`
  - Spring Framework: `org.springframework`
  - Hibernate SQL logging: `org.hibernate.SQL`
  - Hibernate bind parameter logging: `org.hibernate.orm.jdbc.bind`
  - Liquibase: `liquibase`

---

## 4. Current log level strategy

The current configuration is designed to be useful for local development:

- `com.dinobugger.cleanproject` → `INFO`
- `org.springframework` → `INFO`
- `org.hibernate.SQL` → `DEBUG`
- `org.hibernate.orm.jdbc.bind` → `TRACE`
- `liquibase` → `INFO`

### Why this is useful

- you can see application startup clearly
- you can inspect SQL generated by Hibernate
- you can see parameter binding when debugging database issues
- you can watch Liquibase migration output during startup

If SQL output becomes too noisy, lower the Hibernate logger levels later.

---

## 5. Liquibase note

Liquibase logging has **two different parts**:

1. **Application runtime logs**
   - controlled by `log4j2-spring.xml`

2. **Gradle Liquibase task logs**
   - controlled by the Liquibase Gradle plugin configuration in `build.gradle`

These are separate.

That means:

- changing Log4j2 does **not** change the Liquibase migration logic
- changing the Liquibase plugin `logLevel` does **not** change application logging

## 6. How to run and verify

### Start the app

Use your normal Spring Boot start command, for example:

```powershell
.\gradlew.bat bootRun
```

### Run tests

The test profile still works the same way:

```powershell
.\gradlew.bat --% -Dspring.profiles.active=test test
```

You should see Log4j2 output in the console instead of Logback output.

---

## 7. How to test Log4j2

The quickest way to confirm Log4j2 is working is to run the application and look at the console output.

### 1. Test with `bootRun`

Run:

```powershell
.\gradlew.bat bootRun
```

Success looks like this:

- the console log format matches `log4j2-spring.xml`
- the level, timestamp, and class name appear in the output
- application startup messages are printed by Log4j2, not Logback

### 2. Test with the `test` profile

Run:

```powershell
.\gradlew.bat --% -Dspring.profiles.active=test test
```

Success looks like this:

- the tests still pass
- Log4j2 output appears during startup and test execution
- Liquibase and Hibernate logs follow the configured levels

### 3. Optional quick proof with a temporary log statement

If you want a very direct check, add a temporary `log.info("Log4j2 is working")` in any class and run the app again.

If the message appears with the Log4j2 pattern, the setup is correct.

### 4. If no Log4j2 output appears

Check these items first:

- `spring-boot-starter-log4j2` is present
- `spring-boot-starter-logging` is excluded
- `src/main/resources/log4j2-spring.xml` exists and is loaded

## 8. Troubleshooting

### 1. You still see Logback-style logs

Check that:

- `spring-boot-starter-log4j2` is present
- `spring-boot-starter-logging` is excluded
- `log4j2-spring.xml` is located under `src/main/resources`

### 2. No logs appear

Check that:

- the console appender exists in `log4j2-spring.xml`
- the root logger is configured
- the application is not redirecting output elsewhere

### 3. SQL logs are too noisy

Reduce these logger levels in `log4j2-spring.xml`:

- `org.hibernate.SQL`
- `org.hibernate.orm.jdbc.bind`

### 4. Liquibase output is missing

Check the `liquibase` logger level in `log4j2-spring.xml` and the `logLevel` setting in the Gradle Liquibase activity.

## 10. Summary

This project now uses Log4j2 as the application logging backend.

The essential pieces are:

1. add `spring-boot-starter-log4j2`
2. exclude `spring-boot-starter-logging`
3. keep `src/main/resources/log4j2-spring.xml` in place
4. adjust logger levels in that file when needed

That is all you need to get started with Log4j2 in this repository.
